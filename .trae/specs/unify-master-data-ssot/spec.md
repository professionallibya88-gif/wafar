# مواصفة توحيد البيانات الأساسية (SSOT) Spec

## Why
البيانات الأساسية (الماركات، الموديلات، القوائم المرجعية، ومسارات الأصول) متفرقة بين عدة أماكن، وهذا يسبب تعارضات في القوائم المنسدلة وأخطاء ربط تؤثر على الاعتمادية. نحتاج مصدر حقيقة واحد قابل لإعادة الاستخدام بين الواجهة والخلفية مع الحفاظ الكامل على واجهة المستخدم.

## What Changes
- إنشاء وحدة مركزية للبيانات الأساسية داخل المستودع لتكون المصدر الوحيد للقوائم الثابتة.
- تعريف واجهات وأنواع TypeScript صارمة للبيانات المرجعية وتصدير موحد عبر `index.ts`.
- إضافة سجل موحد لمسارات الأيقونات/الأصول مع قيمة افتراضية fallback لمنع كسر العرض عند فقدان أصل.
- إضافة مسار API عام `GET /api/v1/master-data` يعيد البيانات الموحّدة بصيغة JSON مهيكلة.
- ربط مكونات الواجهة التي تستخدم قوائم ثابتة لاستهلاك المصدر المركزي (استيراد مباشر أو جلب عبر fetch نسبي).
- إزالة مصادر البيانات المحلية المكررة بعد إتمام الربط والتحقق.
- **BREAKING**: أي استيراد مباشر قديم من ملفات بيانات محلية سيُعتبر غير معتمد ويجب تحويله إلى المصدر المركزي.

## Impact
- Affected specs: القوائم المنسدلة في البحث والفلاتر، تكامل بيانات النماذج المرجعية، استقرار عرض الأيقونات/الشعارات.
- Affected code: `apps/frontend/src/**`, `apps/backend/src/routes/**`, `apps/backend/src/services/**`, ومجلد مشترك جديد للـ master-data داخل المشروع.

## ADDED Requirements
### Requirement: Master Data Single Source of Truth
النظام SHALL يوفّر وحدة مركزية واحدة تحتوي القوائم الأساسية (الماركات، الموديلات، الأنواع المرجعية، وسجل الأصول) مع أنواع صارمة وتصدير موحّد.

#### Scenario: Central data is consumed consistently
- **WHEN** أي مكون أو خدمة يحتاج بيانات مرجعية
- **THEN** يستهلكها من الوحدة المركزية فقط دون نسخ محلية بديلة

### Requirement: Public Master Data Endpoint
النظام SHALL يوفّر مسارًا عامًا `GET /api/v1/master-data` يعيد البيانات الأساسية من المصدر المركزي مباشرةً.

#### Scenario: Client fetches master data
- **WHEN** يستدعي العميل `fetch('/api/v1/master-data')`
- **THEN** يحصل على JSON مهيكل يحتوي جميع الحقول المرجعية المتفق عليها بدون الحاجة إلى صلاحيات

### Requirement: Asset Mapping Fallback Safety
النظام SHALL يوفّر آلية fallback لمسارات الأصول داخل سجل موحد لمنع ظهور أيقونات/شعارات مكسورة.

#### Scenario: Missing asset mapping key
- **WHEN** يطلب المكون مفتاح أصل غير موجود
- **THEN** يعيد النظام المسار الافتراضي المحدد بدل قيمة فارغة أو مكسورة

## MODIFIED Requirements
### Requirement: Frontend Dropdown Data Binding
ربط القوائم المنسدلة SHALL يعتمد على مصدر مركزي موحد بدل البيانات المكررة داخل كل شاشة.

#### Scenario: Existing filter component loads options
- **WHEN** يفتح المستخدم أي فلتر يعتمد ماركة/موديل/نوع
- **THEN** تظهر القيم الصحيحة من SSOT بنفس الـ props الحالية وبدون تغيير في شكل UI

### Requirement: Integration Safety Without UI Changes
أي تحديث لطبقة البيانات SHALL لا يغيّر التصميم أو الألوان أو التخطيط أو سلوك التفاعل المرئي.

#### Scenario: Refactor completes
- **WHEN** تنتهي عملية توحيد البيانات
- **THEN** تبقى تجربة الاستخدام بصريًا وسلوكيًا مطابقة قبل/بعد التغيير

## REMOVED Requirements
### Requirement: Local Duplicated Master Data Files
**Reason**: تسبب تضاربًا في القيم وصعوبة صيانة وتكرارًا غير موثوق.
**Migration**: تحويل جميع الاستيرادات إلى الوحدة المركزية أو مسار `GET /api/v1/master-data` ثم حذف الملفات المحلية المكررة.
