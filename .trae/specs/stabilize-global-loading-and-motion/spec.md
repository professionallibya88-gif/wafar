# نظام الاستقرار البصري والانتقالات الموحد Spec

## Why
المشروع يعاني من مشاكل عامة ومترابطة في أول تحميل، وميض الثيم، تأخر الخلفيات البصرية، وعدم اتساق الانتقالات بين الصفحات والأقسام. المطلوب ليس ترقيعات موضعية، بل نظام معماري موحد يمنع الشاشة الفارغة والوميض والاهتزازات البصرية ويمنح تنقلاً رسمياً هادئاً ومتقدماً على مستوى المشروع كله.

## What Changes
- إنشاء طبقة Bootstrap Shell موحدة تظهر فوراً قبل أول رسم للتطبيق وتمنع الشاشة الفارغة.
- نقل تهيئة الثيم إلى مرحلة مبكرة قبل `mount` لمنع وميض `dark/light` وتبدل الخلفيات.
- إعادة تنظيم تحميل الإعدادات العامة والخطوط بحيث لا تسبب إعادة رسم عنيفة أو تأخر بصري.
- توحيد نظام الانتقالات بين الصفحات على مستوى `App` و`layouts` و`routes` بسياسة مركزية واضحة.
- اعتماد سياسة تحميل موحدة تميز بين:
  - التحميل الأولي للتطبيق
  - تحميل route
  - تحميل بيانات الصفحة
  - العمليات الحاجبة القصيرة
- تحويل الخلفيات الثقيلة بصرياً مثل مشهد المصادقة إلى Progressive Enhancement يبدأ بخلفية ثابتة ثم يعزز لاحقاً بدون وميض.
- إضافة ضوابط هندسية للأداء وإمكانية الوصول مثل `prefers-reduced-motion`.

## Impact
- Affected specs: تجربة الإقلاع الأولى، الثيم، الانتقالات، صفحات المصادقة، حالات التحميل، استقرار الواجهة البصرية.

## ADDED Requirements
### Requirement: Bootstrap Shell موحد
يجب أن يوفر النظام طبقة تحميل أولية موحدة تظهر فوراً قبل أول Route وتبقى حتى تصبح الواجهة جاهزة للرسم الأول.

#### Scenario: فتح التطبيق لأول مرة
- **WHEN** يفتح المستخدم المنصة أو يعيد تحميل الصفحة
- **THEN** تظهر طبقة Bootstrap Shell متوافقة مع الثيم وتمنع ظهور شاشة فارغة أو سوداء قبل جاهزية التطبيق

### Requirement: تهيئة ثيم مبكرة
يجب أن يحدد النظام الثيم الفعلي ويطبقه قبل أول رسم مرئي للتطبيق.

#### Scenario: مستخدم يملك ثيم محفوظ
- **WHEN** يزور المستخدم المنصة ولديه ثيم محفوظ أو ثيم نظام
- **THEN** تُرسم الصفحة منذ البداية بالثيم الصحيح بدون وميض أو تبدل لوني بعد الظهور

### Requirement: نظام تحميل موحد متعدد الطبقات
يجب أن يعتمد المشروع سياسة تحميل معيارية تفصل بين تحميل التطبيق، وتحميل التنقل، وتحميل البيانات، والعمليات الحاجبة.

#### Scenario: تحميل صفحة بيانات
- **WHEN** ينتقل المستخدم إلى صفحة تعتمد على API
- **THEN** لا تظهر دائرة وسط الشاشة كحل افتراضي، بل تُستخدم Skeletons أو Route Shells بحسب نوع الحالة

### Requirement: انتقالات صفحات مركزية
يجب أن تُدار انتقالات الصفحات والأقسام من نظام مركزي موحد يدعم RTL ويحدد نوع الانتقال عبر سياسة واضحة.

#### Scenario: التنقل بين صفحة وقسم فرعي
- **WHEN** ينتقل المستخدم بين صفحة رئيسية وصفحة تفصيلية أو يعود للخلف
- **THEN** يطبق النظام انتقالاً متسقاً وهادئاً يطابق اتجاه التنقل والسياق الوظيفي

### Requirement: تعزيز تدريجي للمشاهد البصرية الثقيلة
يجب ألا تعتمد الصفحات الثقيلة بصرياً على محرك الرسم المتقدم كشرط لظهور أول واجهة.

#### Scenario: فتح صفحة تسجيل الدخول
- **WHEN** يتم فتح صفحة المصادقة
- **THEN** تظهر الخلفية الأساسية والواجهة فوراً، ثم تُفعّل المؤثرات الثقيلة لاحقاً بدون وميض أو قفز بصري

### Requirement: دعم تقليل الحركة
يجب أن يحترم النظام إعدادات المستخدم الخاصة بتقليل الحركة.

#### Scenario: تفعيل تقليل الحركة من النظام
- **WHEN** يكون `prefers-reduced-motion` مفعلاً
- **THEN** يخفف النظام الانتقالات والحركات غير الضرورية مع الحفاظ على وضوح حالات التحميل والتنقل

## MODIFIED Requirements
### Requirement: سياسة الحراسة والتنقل الأولي
يجب تعديل سلوك حراسة الراوتر بحيث لا تمنع أول رسم مرئي إلا عند الضرورة الأمنية الفعلية، مع ترحيل التحميلات غير الحرجة إلى ما بعد ظهور الـ Shell أو الصفحة الأساسية.

### Requirement: صفحات المصادقة
يجب أن تعتمد صفحات المصادقة نفس النظام المركزي للتهيئة والانتقالات والاستقرار البصري، وألا تستخدم سلوكاً انتقالياً منفصلاً أو خلفية تتأخر عن الواجهة الأساسية.

### Requirement: تطبيق الخطوط والإعدادات العامة
يجب أن يُدار تحميل الخطوط والإعدادات العامة بطريقة تمنع إعادة رسم شاملة وغير مستقرة بعد ظهور التطبيق.

## REMOVED Requirements
### Requirement: مؤشرات تحميل غير مهيكلة كحل افتراضي
**Reason**: الاعتماد على Spinner عام أو انتظار صامت لا يحل مشاكل الشاشة الفارغة أو الوميض أو عدم الاستقرار في أول تحميل والتنقل.
**Migration**: استبدال هذه السياسة بطبقات موحدة تشمل Bootstrap Shell وRoute Shell وSkeleton وBlocking Overlay وفق سياق الاستخدام.
